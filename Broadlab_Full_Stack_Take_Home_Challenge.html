Full Stack Engineer Take-Home Challenge:
The Campaign Dashboard

Estimated time commitment: please spend no more than 1–2 hours on this exercise.

At Broadlab, we help agencies and brands make Addressable TV advertising smarter. Our platform brings together audience intelligence, campaign planning, supply allocation, and performance measurement in one place.

We want to see how you approach building a small but complete feature — from data to API to UI — using tools close to our actual stack.

Your Mission

We've provided a sample dataset of campaign delivery data (30 days of impressions, spend, and revenue across 26 deals). Your task is to build a simple Campaign Performance Dashboard with a Python backend and an HTML/JS frontend.

What to Build

1. A Python/Flask API (required)

Create one or more API endpoints that serve useful summary data from the provided dataset. For example:

  • Total impressions, spend, and revenue by buy_type
  • Daily trends (e.g. impressions over time)
  • Margin by deal or buy type (gross_revenue - spend)
  • Top/bottom performing deals

You decide what's useful — we want to see your judgement about what data matters, not a prescribed list.

2. A simple frontend (required)

Build an HTML page that fetches from your API and displays the data meaningfully. This should include:

  • At least one summary table
  • At least one chart or visualisation (any JS charting library is fine — Chart.js, Plotly.js, D3, or even a simple canvas drawing)
A note on technology choices: Our current frontend is vanilla JavaScript (no framework) with Tailwind CSS. You're free to use whatever frontend approach you prefer for this exercise — vanilla JS, React, Vue, or anything else. Same for CSS — Tailwind, Bootstrap, or plain CSS are all fine. We'll ask you to explain your choices in the follow-up conversation, so be ready to talk about why you picked what you picked.

3. One interactive element (required)

Add at least one user interaction, for example:

  • A filter (by date range, buy type, or deal)
  • A sort toggle on the table
  • Click a row to see detail

4. Deployment sketch (optional, written only)

In a few sentences, describe how you would deploy this to AWS. Don't build it — just tell us what services you'd use and why.

The Data

We've included the data as a JSON file:

  • campaign_delivery_sample.json

Each row represents one deal's delivery for one day, with fields:

Field Description
dateDelivery date (YYYY-MM-DD)
deal_idUnique deal identifier
buy_typeChannel category (CTV, Display, Audio, BVOD)
supply_partnerThe supplier/platform
channelDelivery channel
impressionsNumber of ad impressions delivered
spendSupply cost (what we pay the partner)
gross_revenueGross revenue (what the client pays us)
is_measurableWhether the deal has measurement capability

This is fictional data — no real campaigns or partners are represented.

What We're Looking For

  1. Working software. It should run. A simple, working app beats an ambitious broken one.
  2. Code quality. Clean, readable, well-structured. We'll read your code — make it easy to follow.
  3. Judgement. What did you choose to show and why? We care about your thinking, not just your output.
  4. Frontend fundamentals. Layout, interaction, data display — and the reasoning behind your technology choices.
  5. Backend fundamentals. Is the API well-designed? Sensible routes, clean data handling.
A note on AI tools: You're welcome to use AI-assisted development tools (Copilot, Claude, ChatGPT, etc.) — we use them daily and consider them part of modern engineering. However, you must understand every line of code you submit. We will discuss your approach and decisions in a follow-up conversation, and we'll ask you to explain specific choices. "The AI suggested it" is not an acceptable answer.

Deliverables

Submit your work as a Git repository (GitHub, GitLab, or a zip). Include:

  • Your Python/Flask backend code
  • Your HTML/CSS/JS frontend code
  • A brief README.md with setup instructions and any notes on your approach
  • The deployment sketch (if included) can go in the README

What Not to Worry About

  • Authentication or user management
  • Database setup — loading from the JSON/CSV file is fine
  • Production deployment — local development is all we need
  • Pixel-perfect design — functional and clear beats beautiful and broken
